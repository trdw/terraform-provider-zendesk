# Zendesk Provider

The Zendesk provider manages configuration objects in a Zendesk Support instance via the [Zendesk REST API](https://developer.zendesk.com/api-reference/ticketing/introduction/). Every supported object has both a resource (for managed configuration) and a data source (for looking up existing objects by id or natural key).

## Example Usage

```terraform
terraform {
  required_providers {
    zendesk = {
      source = "yourorg/zendesk"
    }
  }
}

provider "zendesk" {
  subdomain = "my-organization"
  email     = "admin@example.com"
  api_token = var.zendesk_api_token
}

resource "zendesk_group" "support" {
  name = "Support"
}
```

Credentials can also be supplied via environment variables: `ZENDESK_SUBDOMAIN`, `ZENDESK_EMAIL`, `ZENDESK_API_TOKEN`.

## API Quota & Backoff

Zendesk enforces a per-account API rate limit that varies by plan and endpoint. See [Zendesk's rate-limit reference](https://developer.zendesk.com/api-reference/introduction/rate-limits/) for the current ceiling on your account. Every API call also competes with traffic from agents, integrations, and other automation, so a Terraform plan/apply with hundreds of resources can both hit the ceiling and starve the rest of the system.

The provider tries to be a good neighbor without making you tune anything:

- **Headers tracked.** After every response the provider records `X-Rate-Limit` (total) and `X-Rate-Limit-Remaining` / `ratelimit-remaining` plus `ratelimit-reset` so it always knows what fraction of the quota is left and when the window resets.
- **Pre-emptive throttling.** Below 50% usage, no delay. Between 50% and 90% the provider sleeps before each request, growing exponentially from 100ms up to 60s as usage approaches 90%. At or above 90% the provider blocks until the rate-limit window resets. This smooths out a burst long before Zendesk has to push back, and leaves headroom for everything else hitting your account.
- **429 retries.** If a request still gets `HTTP 429 Too Many Requests`, the provider honors the `Retry-After` header (or falls back to 60s) and retries the request up to three times. After that the operation fails with a clear rate-limit error.

These delays are emitted at `[DEBUG]` (throttling) and `[WARN]` (critical wait, 429 retry) — set `TF_LOG=DEBUG` or `TF_LOG_PROVIDER=DEBUG` to see them. The throttling and retry behavior are not user-configurable today; if your workflow needs different thresholds, open an issue.
